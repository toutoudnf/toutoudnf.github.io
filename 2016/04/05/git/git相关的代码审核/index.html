git相关的代码审核

发表于 2016-04-05 | 分类于 git

记录了最近做代码review的相关心得。

review的契机

代码review真的是很重要的东西。

像给老左review scheduler module，确实是review出来问题的。

另外review的时候给别人讲自己的代码，也是有用的。不知别人如何想，自己讲的时候也是能发现一些坑的。

review过程中的总结

review的规范

粒度不必太细

庆龙定了个很详细的标准，但是用起来挺别扭的，因为记不住。

功利一些的想法，我们要体现出review的价值，其实可以考虑从这个角度出发来定标准。没有目的性的review一般会做无用功。就是这个事儿要办成一个可量化的事儿，埋点加指标，出报告。要跟上面leader说，这个费力费人的事儿，到底做了啥。

所以当时提了个简单的分类方案：

  • bug
  • 实现优化
  • 注释

这个类别可以调整，但是增加而不要删除，防止之前的

review的方式

review时候的目的性+方式

  • review没必要是两个人，可以reviewer自己看，看不懂再问
  • review得有个统一的目的性，比如review Hibernate的用法，或者看sql写法balabala
    • 突然想到，如果review能结合一些培训，可能效果会更好的
    • 因为需要review，除了个人会犯错这种情况外，更多的是对被review人的技能缺失/思维逻辑不足而review

gerrit

gerrit挺好

其实工具的重要性无外乎是干一件事儿省事儿，工具能干的人都能干，效率不同而已。

  • 有一套标记规则
    • 不要复杂，要不记不住
    • 不跟代码重复，能快速检索
  • 能快速检索。即工具可以提供查询，能对review过程中，在代码里面添加的注释进行检索，方便讨论。
  • 能增加备注
    • 这个指的是gerrit中的comment
    • 这个东西用起来，最大感觉是减少人类间沟通，写详细了完全可以在上面完成80%的问题交流，不用面对面去说问题了。
    • 是对一个change中提交的代码问题的标注
  • topic这个东西其实挺坑的，二次review -d的时候会覆盖掉

测试先行的重要性

review是个长期持续的事儿，故是比较依赖测试先行的。

测试先行的模式能提供重构的可能，这是好多事儿的一个基础吧。目前比较蛋疼的事儿就是不敢改。

持续性的测试

测试的严格程度+reviewer的水平不可能是一致的，也不可能是最严厉的。因为要考虑每个人的水平是不一样的。所以这必然是个长期而持续的事儿，甚至可能一些最牛逼的人在一开始也发现不了的问题，会埋在代码中。故针对这种长期战斗，有一个查缺补漏的流程比较重要。

啥是查缺补漏呢，就是我今天发现一个Hibernate的严重用法问题，那么我要有个机制能迅速发起一个指定主题的review。然后与常规的review结合，效果会好。

review的自动化

目前看到的，基本是codestyle的check吧，能替代一部分的review工作，不过有限。
