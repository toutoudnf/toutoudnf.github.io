Rebase能做啥

  • 不产生新的commit情况下，合并两个branch
  • 能保持branch log的线性
  • 但是rebase可能会造成之前的commit历史被更改
  • 其实本质呢，可以理解为把一个commit集合在指定commit上做rebase
    • –onto指定了一个commit，即target commit
    • 需要一个commit集合
      • git rebase master develop
      • 等价于
      • git rebase –onto master master..develop
  • 上面的本质呢那个啥啥啥的不对，验证过了master..develop这种写法is wrong
  • so 别信啊先，等我搞清楚

Rebase details

rebase参数

  • rebase –continue
    • 解决冲突后，使用；rebase是一个持续的过程
  • rebase –skip
    • 跳过当前patch；本质讲，一个commit就是一个patch嘛
  • rebase –abort
    • abort当前rebase，并回复；HEAD也back to origin branch
  • rebase –edit-todo
    • Edit the todo list during an interactive rebase

rebase case

git rebase

  • 没参数时，如果指定了upstream，那么会自动rebase remote repository

    git rebase master

  • 等价于下面的case
  • 本质就是把 master..develop这些commit，给挪到了master之后

    git rebase master develop

  • 参见上面

    git rebase –onto master server client

  • 取出 client 分支，找出处于 client 分支和 server 分支的共同祖先之后的修改，然后把它们在 master 分支上rebase一遍
