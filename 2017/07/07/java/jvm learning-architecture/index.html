JVM 架构学习的笔记。

  • 了解 JVM 有哪几部分组成，以及对应的职责

JVM 整体架构

来个 JVM 整体架构图镇楼：

主要包括以下三部分：

  • garbage collerctor：负责收集 runtime 区域的用不到的内存
  • JIT compiler：负责动态优化
  • HotSpot VM runtime：负责 vm runtime 内存管理

需要注意的是，gc 模块与 JIT 模块都是可插拔的。这两个模块都是调用 VM runtime 模块的 API 接口来实现自己的功能。

这一张不会涉及太多的技术细节，而是想从概念上梳理清楚，JVM（HotSpot为例）到底有哪几部分组成，分别负责什么。下面会参照着三部分开始介绍。

runtime

runtime 模块是一个 JVM 的核心，除了为 gc 和 JIT compiler 提供相应的接口之外，他还提供了：

  • 启动
  • 线程管理
  • JNI

等等功能。组件很多，学习很复杂，那么我们从一个应用运行的角度来考虑其职责，并按照这条线梳理。一个正常的应用的大致完整的声明周期大概可以分为，大概要经过：

  • 开始
  • 启动一个可提供服务的 JVM（runtime 部分）
  • 加载应用运行所需的类
  • 找到启动类（main），并执行其中相应逻辑（加载新的类，or 执行某个方法）
  • 被人为终止或者接收到了中断信号，VM 停止运行，并释放对应的资源（内存等）
  • 结束

ok，那根据上述描述，可以简单假设，VM 需要：

  • 一个启动器，完成 VM 自身启动
  • 类加载器，完成应用所需的类的加载
  • 需要一个将 class 字节码翻译为机器码的组件（解释器）
  • 执行代码
    • 需要创建对象的时候，需要申请内存使用（内存管理）
    • 程序是多线程的，那么怎么映射到具体的操作系统的线程呢？（线程管理器）
  • 如果出现重大异常，或者人为调用退出接口，那么清理掉内存中的类（启动器的销毁方法）
  • 运行结束，释放 VM 占用的资源（内存，线程等）

除此之外，考虑到对 VM 性能的优化，一般在启动之前可以指定一些参数，来对 VM 的初始化，运行做一些优化。这就是参数机制咯。基本上上面提到的加粗的部分，是一个 VM runtime 的主要组件了。更详细的内容会在接下来的文章中讲到。

gc

之所以出现垃圾收集器，是基于 JVM 在内存管理中的以下几个特点：

  • 大多数分配对象存活时间短
  • 存活时间久的对象，很少引用存活时间段的对象
  • Java 语言大部分情况的内存管理是由 JVM 完成的，所以存在内存释放的需求

上述的 1，2 两条有个名词儿叫弱分代假设（weak generational hypothesis）。为什么会得出这么一个假设，可以通过简单的对一个典型的 Java web app 来进行一些分析：

  • 存在一些全局变量（比如使用 spring 去管理的 bean 对象，静态常量类等）
  • 大部分情况下是在处理一个 Http 请求，对象的生命周期基本为：session 级别；request 级别；具体到每个方法中的局部变量 level。这几种对象的特点是：
    • 在内存中存活时间短
    • 一般都是他们去引用全局变量做处理，而不是全局变量去引用他们

由此而得，跟前面的弱分代假设还是比较接近的。可以参考应用的实际情况做一些分析。

既然对象存在这样的特点，那么统一扫描，判断，然后处理对象是否需要回收，显然是不合理的。于是 HotSpot 将 VM 中的堆（对象主要存放的地方），划分为：

  • 新生代：比较小，其中的对象回收频繁，并且对象回收的比例高。对新生代执行的垃圾收集有个专门的名词叫 minor gc。
  • 老年代：比较大，其中的对象由新生代晋升而来（新生代多次扫描都干不掉的），一般是生命周期较长的对象，故扫描比相比新生代频率要低。对老年代的垃圾收集有个专有名词叫 full gc。

另外，针对一些 VM 的元数据（如类数据结构，保留字符串等），这种对象有个特点是：

  • 大小不会变化太多
  • 生命周期很长，并且一般可预见

这样的对象即使放到老年代也是比较浪费扫描操作的，故 VM 划分了永久代来管理这些对象。需要注意的是老年代对象即使过了很久很久没被收集也不会晋升到永久代。

上面就是 HotSpot 最基本的内存管理模型，既然不同类型对象做了区分，那么针对不同区域的堆，gc 模块也提供了不同的回收机制。

  • serial
  • throughput
  • CMS
  • G1

是不是有些眼熟，在各种博客上可能经常看到这些名词。这里不做详细介绍，只要知道目前主流 gc 有大概几种即可。

JIT

JIT 属于 VM 性能优化的一部分，在后面会详细介绍下。在学习之初，了解如下相关 JIT 的特性就足够了：

  • JIT 分文 server 和 client 两种模式，他们各有侧重，分别为：
    • client JIT 编译器目标是更快的启动时间和快速编译
    • server JIT 编译器目标是 Java 应用性能优化

如果从学习 VM 并优化 VM 性能的角度来看，JIT 相关的优化点可能要放到 dump，代码等等其他方式之后。甚至是机器之后。就像 MySQL 的 SQL 执行计划一样，内部的优化机制可能在每个版本的 Java 应用都不相同，并且甚至是不兼容的。应该是在其他方法都尝试了并且确实有需求的情况下，再根据当前版本的 JIT 特性来有针对性的进行代码或者其他方面的优化。

简单总结

目前为止，我认为学习 JVM 开始，对于架构方面有一个基本的了解即可。细究不是不好，是会耽误后面学习的一些进度，并且对于我这种比较笨的选手来说，不明确思路的学习可能会事倍功半。学习完后尝试提问自己几个问题：

  • HotSpot 为例，JVM 有哪几部分，职责是什么？
  • 如果让你来实现一个 VM 的话，runtime 应该有些什么，而 HotSpot VM 中包含了哪些基本组件？本文中涉及到的可能缺失什么呢？
  • HotSpot 的基础堆管理模型是什么？为什么？
  • HotSpot VM 的 gc 有哪几类（这貌似是个填鸭题，笑：）？
