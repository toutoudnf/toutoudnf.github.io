RESTFul的一些讨论

发表于 2017-02-09 | 分类于 servers

今天开会讨论了Hans的需求分析过程&对应文档中存在的问题，其中跟庆龙讨论RESTFul部分相关的内容感觉很棒，记录如下。

今天的流程

会议讨论的主要是文档相关的内容，中间包含了一部分关于需求文档如何转化为RESTFul接口的讨论。下午相关的讨论了一些hans现在RESTFul api中存在的一些问题，特记录如下。

需求文档===》RESTFul api===》流程图

解释一下这个过程。

首先是对需求文档，按照业务进行模块划分。划分的时候参考了openstack的结构以及build14现有的需求功能，层级分为两个level。这里的level主要用于对应之前的需求使用，意义不大。

ok那么怎么从需求文档拆解到流程图呢？流程图的概念等价于接口设计（详细设计level），即接口间的调用。梳理到这个level主要是：

  • 减少实际开发工作量。这个level已经能完成部分的代码开发工作了。
  • 模块解耦，这就是back end R&D间的交流文档

针对hans的后端研发（面向FE&SDK的开发），RESTFul api是我们的最终交付物。需求文档中的需求，每个部分都要拆解到一个对应的api。中间可能会遇到的问题：

  • api重复
  • 非功能需求

这些需要单独的列出来。然后对全部的api进行合并去重。这样会得到一个controller level的，全部需要开发的接口清单。接下来把这些URL向规划好的ROOT api tree上去套，如/*version*/*module*/specified_url的结构。这个过程本质其实是URL语义（叶子结点的需求）到具体api URL的转换过程。然后针对每个api，绘制流程图，完成controller==》business==》dao三层的接口调用的分析。

哈，大概过程是这样，讨论了不少，最后想明白以后还是蛮简单的。

一些结论

讨论过程针对现有RESTFul的一些坑&注意点，记录如下：

存在的坑

实现与语义不符。

如/hosts/{id}/adapters/{id}/disks/{id}的情况，需要对三个id均进行validate，才是合理的。

不要care 资源处于顶级level是否合理

如今天讨论task&notification的时候，放顶级level觉得不合适，但是归到哪种资源下也不合适。这些需要的就是一个虚的parent，如system managment之类的，不要再这些URL总量占比小的资源类别上花过多的心思。

比较难处理的

URL语义中动词部分的处理

这个总结了下，大概是两种思路。以adapter的scan举例：

  • scan名词化。变为/adapters/{id}/scanistor，用PUT触发
  • scan参数化。/adapters/{id}?action=scan

同一个需求，有多个URL实现。

即同样一个disk查询，我既可以/disks/{id}，也可以/hosts/{id}/disks/{id}。那么这种情况怎么办？原则上，是按照实际的需求为准。这个需求可能包括：

  • 业务功能的需求，需要考虑SDK/FE等不同的需求
  • 自动化测试需求

业务术语的规范化

如action，如各种资源都会用到的state，status，type等，在restful api中的语义。
